# Modular Operational Surface System

Status: Active
Date: 2026-05-17
Audience:
- frontend engineering
- UX architecture
- product design
- operational systems engineering

---

# Purpose

This document defines the modular operational surface system that should replace the current box-heavy dashboard approach.

The current UI still behaves largely like:

- enterprise dashboards;
- stacked operational cards;
- fragmented widgets;
- disconnected operational summaries.

The next IndiCare UI should instead behave like:

> adaptive operational surfaces.

Inspired by:

- macOS widgets;
- Apple Smart Stacks;
- VisionOS spatial modules;
- Linear workspace surfaces;
- Arc Browser spaces.

---

# Core principle

Operational surfaces should feel:

- alive;
- contextual;
- adaptive;
- spatial;
- calm.

NOT:

- bolted onto pages;
- visually equal;
- rigid;
- analytics-heavy.

---

# Why the current model feels wrong

The current UI still has:

- too many equal-weight rectangles;
- too many borders;
- too many competing surfaces;
- weak spatial hierarchy;
- operational duplication.

This creates:

- cognitive fatigue;
- visual flattening;
- enterprise dashboard feel;
- operational overwhelm.

---

# Strategic direction

The platform should increasingly render:

- fewer surfaces;
- richer surfaces;
- contextual operational modules;
- larger narrative spaces.

The UI should feel:

> intentionally composed.

Not automatically generated.

---

# Surface categories

## 1. Focus surfaces

Large primary operational surfaces.

Examples:

- Today’s operational awareness;
- Welcome/handover;
- Child overview;
- Significant chronology;
- Open safeguarding follow-up.

Characteristics:

- large;
- spacious;
- minimal distractions;
- emotionally grounding.

There should only be a small number visible at once.

---

## 2. Context surfaces

Secondary operational context.

Examples:

- Notifications;
- Connect previews;
- Upcoming reviews;
- Assigned actions;
- Evidence gaps.

Characteristics:

- lighter weight;
- collapsible;
- expandable;
- reorderable.

---

## 3. Ambient surfaces

Low-priority operational awareness.

Examples:

- subtle chronology indicators;
- relationship indicators;
- continuity prompts;
- wellbeing reminders.

Characteristics:

- visually quiet;
- soft emphasis;
- informational rather than dominant.

---

# Surface behaviour model

Operational surfaces should support:

- collapse;
- expand;
- focus mode;
- pinning;
- contextual resizing;
- soft transitions.

BUT:

Avoid:

- unrestricted drag chaos;
- widget sprawl;
- dashboard fragmentation.

---

# Smart Stack philosophy

The dashboard should increasingly support:

> operational stacks.

Meaning:

related operational items combine into:

- one richer surface;
- one operational narrative;
- one contextual area.

Examples:

Instead of:

- safeguarding card;
- missing card;
- chronology card;
- follow-up card.

Create:

> “Children needing attention.”

Then progressively reveal detail.

---

# Surface hierarchy

The platform should clearly communicate:

1. what matters most;
2. what changed recently;
3. what needs action;
4. what context exists.

This should happen visually through:

- spacing;
- typography;
- scale;
- depth;
- motion.

NOT:

through:

- excessive colour;
- flashing alerts;
- dashboard density.

---

# Motion principles

Motion should:

- orient;
- reassure;
- support continuity.

Use:

- subtle fades;
- expansion transitions;
- smooth stacking;
- soft hover states.

Avoid:

- novelty animation;
- aggressive transitions;
- distracting motion.

---

# Child profile surfaces

The child page should behave like:

> a connected person-first workspace.

The first visible surfaces should communicate:

- identity;
- support;
- relationships;
- communication;
- what helps.

Operational risk appears within context,
not as the emotional centre of the page.

---

# Adult workspace surfaces

The adult workspace should feel:

- personalised;
- grounding;
- shift-aware;
- operationally calm.

The first visible surface should usually be:

- welcome/handover;
- operational focus;
- significant changes since last shift.

Not:

- analytics.

---

# Home operational surfaces

The home view should increasingly feel:

> alive and connected.

Not:

> static management reporting.

Surfaces should support:

- continuity;
- safeguarding awareness;
- operational rhythm;
- recent significant changes.

---

# Surface composition rules

Every surface must justify itself.

Ask:

- Does this improve operational understanding?
- Is this duplicated elsewhere?
- Does this improve continuity?
- Does this improve safeguarding awareness?
- Does this reduce cognitive effort?

If not:

remove it.

---

# Relationship to experience bundles

Experience bundles should power surfaces.

Meaning:

- surfaces render aggregated operational truth;
- frontend orchestration decreases;
- operational coherence increases.

This is how the frontend becomes:

- calmer;
- richer;
- more spatial;
- more connected.

---

# ORB relationship

ORB is intentionally different.

ORB remains:

- expressive;
- energetic;
- colourful;
- intelligence-oriented.

Operational surfaces remain:

- calm;
- grounded;
- spacious.

The contrast is intentional.

---

# Final principle

The future IndiCare interface should increasingly feel like:

> a living operational environment composed of intelligent spatial surfaces.

Not:

> enterprise dashboard software.
