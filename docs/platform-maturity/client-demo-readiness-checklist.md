# Client demo readiness checklist

Use this before any serious provider demo.

## Access and shell

- [ ] Login works for the intended role.
- [ ] MFA behaviour is expected and explained.
- [ ] Command Centre opens after login.
- [ ] No legacy shell route is used as the primary journey.
- [ ] No console errors appear during the demo path.

## Two-minute understanding

- [ ] Command Centre explains what needs attention now.
- [ ] Young person overview explains the child within two minutes.
- [ ] Home overview explains the home within two minutes.
- [ ] Staff can see the primary action without hunting.
- [ ] Mobile layout keeps critical actions visible.
- [ ] Keyboard navigation reaches search, primary actions, modals and forms.

## Recording workflows

- [ ] Daily note can be created, saved and reopened.
- [ ] Daily note appears in records.
- [ ] Daily note appears in chronology where the backend projection supports it.
- [ ] Incident can be created, saved and reopened.
- [ ] Safeguarding record can be created, saved and reopened or clearly states current transport limitation.
- [ ] Missing episode can be created, saved and reopened or clearly states current transport limitation.
- [ ] Health record can be created, saved and reopened.
- [ ] Education record is configured or marked not yet configured.
- [ ] No page reload or accidental form submission bug occurs.

## Documents and evidence

- [ ] Document templates open.
- [ ] A document can be created from template where live child/template support exists.
- [ ] Draft save is clear.
- [ ] Review state is clear.
- [ ] Sign-off state is clear.
- [ ] Documents link to child/home/staff where backend data exists.
- [ ] Documents link to evidence and inspection readiness where backend data exists.
- [ ] Document upload works or shows "not yet configured".
- [ ] Extraction/review works or shows "not yet configured".
- [ ] Documents can be searched quickly.

## Operational intelligence

- [ ] Chronology is understandable and source-linked.
- [ ] Evidence graph/traversal can be explained.
- [ ] Inspection readiness can be explained without raw JSON.
- [ ] Provider oversight can be shown.
- [ ] Governance/audit/replay can be explained at high level.
- [ ] Assistant/ORB opens and responds safely.
- [ ] Assistant/ORB does not present draft support as fact.

## Product quality

- [ ] No duplicate cards.
- [ ] No placeholder/demo-only areas are visible without labelling.
- [ ] No broken buttons.
- [ ] Copy uses British English.
- [ ] Critical safeguarding language is calm and non-punitive.
- [ ] Design feels premium, simple and operationally useful.
